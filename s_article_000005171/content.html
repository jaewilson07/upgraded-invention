Intro

Domo Forms allows you to create and manage reusable forms that can be integrated with Domo systems. Forms are an essential part of the User Task system in Workflows, which powers the task management system in Queues. You can also use Forms as a standalone feature by integrating Forms with App Studio.

This article describes how to access, configure, and manage Forms in the following topics:

Required Grants

The following grants give user-specific rights to Forms in your Domo environment:

  • Manage Forms — Allows you to perform any action on any form in the instance. This should be granted to superusers, admins, or anyone who helps manage forms.
  • Create Forms — Allows you to create a net new form. This should be granted to anyone who needs the ability to create and add forms to the instance repository.
 
Note: Contact your Domo account team to receive either of these grants for your instance. For information about the available permissions for non-instance admin users, see the Forms Permissions section of this article.

Access Forms

 

 
Important: Workflows and Forms must be enabled for your account. Contact your Domo account team for enablement.

Follow these steps to access Forms:

  1. From the navigation header, select More > Workflows.
    The Workflows landing page displays. 


     

  2. In the left navigation, select Forms.
    The Forms home page displays. 

     

 

 
Note: If the options for   Workflows or  Forms do not display in your navigation, they need to be enabled in your instance. Contact Domo Support  or your account team for assistance.

Home Page

The Forms home page displays a list of all available forms based on your permissions, search and filter tools to explore the list, the + New Form option, and possible actions to take on existing forms. The home page components are described below.

 
 

Back to top

Forms Repository List

The Forms Repository List is the main feature of the home page. Each row in the list represents an individual form. This list of column headers describes the information contained in each column:

  • Title — Displays the title of the form.
  • Last Modified — Provides the date when the form was last updated.
  • Last Published — Provides the date when the form was last published. If the form has not yet been published, a dash (–) displays.
  • Created By — Displays the name of the person who created the form.

Action Menu

Each row in the Forms Repository List includes an  action menu after the Created By column that contains available actions for that form. The following actions are available from the menu:

  • Manage Permissions — Opens the permissions modal. To learn about permissions, see the Forms Permissions heading.
  • Duplicate — Creates a duplicate form that immediately displays at the top of the Repository List.
  • View version list — Opens a page that displays the form's version list. You can view historically published versions of the form and select an older version to make it the latest version.
  • View inputs/outputs — Opens a modal that describes the form's contract. The contract is the definition of which fields accept input, which provide output, which are optional, and which are read-only.

Forms Permissions

Inside Forms, users or groups can receive permissions to perform specific actions. These permissions are granted at a form level. To grant permissions, follow the steps below.

  1. In the Forms Respository list, locate the form for which you want to manage permissions and select the action menu.

     

  2. In the menu, select Manage Permissions. 
    The permissions modal displays. The title of the modal is Share Access with the Form. 


    The permissions modal contains a search bar, a list of People With Access to the form, and a list of People Without Access to the form. By default, the owner of the form always has Admin permission to the form. 

  3. In the search bar, enter the name of a person or group for which you want to edit permissions.
    If the user or group is found in the instance, the name displays at the top of the list of People Without Access. The permissions menu displays next to the user or group name; the default permission is No Access.

  4. Select the menu for the user or group and select the permission you want to grant them. The list of possible permissions (levels of access) includes the following: 

    • Admin — Allows the user/group to perform any action for the form.
    • Delete — Allows the user/group to delete the form.
    • Read — Allows the user/group to view the form.
    • Write — Allows the user/group to edit the form.
    • Share — Allows the user/group to share the form with others and grant the level of access that they themselves have.

                     

  1. Select Save to keep your changes.

Create a New Form

 

 
Note: You must have the Create Forms grant to create a new form. See the Required Grants section of this article for more information.

To create a new form, select + New Form on the home page. If this option is disabled, contact your Domo admin or CSM/AE.

 

The form editor opens. See the next section for information about the form editor.

Form Editor

In the form editor, you can build and configure forms with questions, labels, and parameters. You can add as many questions and sections to a form as needed.

There are two areas in the editor: the header and the canvas. These sections are described below.

Validation

As you create forms, required configurations are highlighted in the form in red to help you identify missing details. You cannot publish a form until you have completed all required configurations.

Back to top

Header

The header at the top of the editor contains the following items:

  • Name and Description fields — Select  Edit by either field to add a name for the form or an optional description. After adding either a name or description, select the field again to edit it.
  • Form Owner and sharing — The form owner's name and avatar display beside the  Share option. Select  Share to open the permissions modal. See Forms Permissions for information about this modal.
  • Preview Mode — Toggling the Preview Mode displays a preview of the form in the canvas area of the editor so you can see how the form will look and behave at runtime. While previewing, you can enter data into the form and try out the different field options you have configured.  

  • Form Settings — In  Form Settings, you can hide the form name and section header details as well as create Task Identifiers to search, filter, and identify a specific task in Queues. Learn about Task Identifiers below.

     

  • View inputs/outputs — Selecting  View inputs/outputs opens the Inputs/Outputs Catalog modal, which displays the form's contract or designated inputs and outputs. The modal has two tabs: Inputs/Outputs and Uncatalogued.

    The Inputs/Outputs tab renders each field in the form as a single line item. The field name and alias display on the left side. The alias name displays as a parameter for the entity that integrates with the form (the entity is either a workflow or a Domo app).

     

    The input/output contract displays on the right side of the tab. A green checkmark means that the field is a parameter in the form contract, while a red X means that the field is not a parameter in the form contract. You can change a field's designation by selecting either of these icons. 

     

    An input field requires the integrating system to pass a mapping or value to the form. This is so that the field pre-populates with the mapping or value data when it is rendered to an end user. 

    An output field passes the data in the field to the integrating system for use. When you create a form, you must determine what data the form requires as input and what data the form provides as output.

  • View version list — Selecting  View version list ​​​​​displays a list of all available versions of the form.

    You can save a form as many times as you want before publishing it. The first version of a form has a Draft status until you publish it, at which point the form has a Published status.

    You cannot change a form's contract after the form has been published. This prevents breaking integrations with other products that use the form. However, after you publish a form, you can change its labels, descriptions, and the order of fields and sections.

    When you save a form after publishing it, a new version is created with a Draft status. If you make changes to this version, the changes do not take effect at runtime until you publish it. You can create as many versions as you want, but you cannot change the contract from the original published form.

    You can view a previous version of the form, or make a previous version the current version, using the action menu for that version in the version list.

  • Save menu — The save menu has three options: Save Draft, Safe Draft As, and Save & Publish.

     

    You can save your form as many times as you want before you publish it. To save a form, select Save Draft in the page header. The save menu options are as follows:

    • Save Draft — Saves the form in a draft state.
    • Save Draft As — Creates a copy of the form as a new form in the Repository List. This works in the same way as the Duplicate option on the Forms home page.
    • Save & Publish — Saves the form and publishes it. You can then select the form for integration  with other Domo products. 
       
      Note: After publishing  a  form, you cannot  change its contract (inputs and outputs). See View version list for more information about publishing.

Canvas

The canvas area of the editor contains the sections of the form with a number of items, or questions, in each section. 

Each section is a different page in the form and can contain multiple items.

Each section displays a section count: the section number out of the total number of sections in the form. In this example, the section is the first of one section.

 
You can minimize or maximize a section of the form by selecting  Collapse or  Expand. If you try to collapse a section without completing all the fields in the section, the form errors and you receive an alert that the Add Name field must have a value.

Select the section action menu to view possible actions for the section: Duplicate section, Move section, and Delete section. You cannot move or delete a section if there is only one section in the form, because a form must have at least one section. This list describes each possible action:

  • Duplicate Section — Copies the section and places it directly below itself.
  • Move Section — Allows you to move the section using drag-and-drop functionality.
  • Delete Section — Removes the section from the form. 



 

For each section in the form, fill in the Add Name field and enter an optional description of the form in the Description field by selecting  Edit beside either field.

Select + Add Section to create a new section of the form. Each section represents a different page of the form when the form renders at runtime, so if you want one page, you only need one section.

 

For each item in a section, there are Question, Question Type, and Data Type fields.

 

 
Required Toggle: The Required toggle allows you to make an item required or optional for form users. By default, this toggle is on, making this item required. Required items force users to enter a value before submitting the form.

Question Label and Description

Enter the text of your question or requested input in the Question field. In this example, the requested input is InputOrderNum.

To add an optional description for the Question field, select + Description (Optional).

Question Type

For each item, you must specify the question type in the dropdown. The type determines how the form displays in the system it integrates with and how the form user populates the field. For example, Short Answer provides the form user with a simple text field, while Date/Time provides a calendar and time picker.

The possible question types are listed below:

  • Multiple Choice
  • Single Choice
  • Short Answer
  • Date/Time
  • Duration
  • Attachment
  • Title/Description
  • List Response
  • Paragraph

Data Type

For each item, you must specify the data type. The Data Type menu filters the accepted data types for your selected Question Type, as described in the previous section.

The possible data types are listed below. Not all data types are available for each question type.

  • Text
  • Date
  • Time
  • Date/Time
  • Number
  • Decimal
  • Duration
  • Person
  • Dataset
  • Group

Add Item 

Select Add Item to create a new item in the section/page. You can add as many items to a section/page as you want.

Duplicate Item

Selecting  Duplicate copies places a duplicate of the item directly below the original item in the same section.

 

Delete Item

Selecting  Delete removes an item from the form.

Reorder Items

Use the handles to drag items and reorder them within their section. You cannot drag items between sections.

Advanced Settings

Each item has advanced settings, which you can reach through the kebab menu at the bottom of the item. When you select this menu, the following options display:

  • Accepts Input — Makes the item an input parameter of the form. The integrating system must provide a data value or values to populate the field when it renders. This method of marking a field as input is in addition to the method described under View inputs/outputs.
  • Provides Output — Makes the item an output parameter of the form. The integrating system receives the data value or values entered into the field after form submission. This method of marking a field as output is in addition to the method described under View inputs/outputs.
  • Read Only — Disables the item from accepting data entered by the form user. You should use this option if you do not want users to change the value displayed in the item.

Define Field Options in External Mapping

Single Choice and Multiple Choice question types allow second input mapping. When you select one of these field types, the Define field options in external mapping checkbox displays.

 

By default, the checkbox is not selected, requiring the form user to enter values in the field. Selecting the checkbox indicates that you are mapping the options from an external source.

 

 

 
Note: When you check the box, any options manually added to the question are deleted, and a warning message displays.
 

Additionally, selecting the checkbox displays the contract modal with external values as an additional parameter for the field. For information about this modal, see the explanation of View inputs/outputs.



When you enable Accept Default Input for these question types, a mapping is required to have one or more values (depending on the field type) pre-selected when the form is rendered to an end user.

You can also preview how the field behaves at runtime.

Task Identifiers

Task identification and retrieval are achieved by displaying specific field values in a task to help users distinguish tasks in the Queues tool.

The Task Identifiers setting allows you to flag specific fields in your form. The Task Identifiers column presents unique data points in the Queues Task List. You can also search these values to filter by specific keywords to enable an improved task management experience.

Follow these steps to create Task Identifiers:

  1. Select   Form Settings in the top navigation of your form.

     
  2. Toggle the Create Task Identifiers option.



    An input section displays so you can enter text or variables from the form.
  3. Select + Add New to enter a new task identifier.
    A new item is added to the list. 
  4. (Optional) Select the dropdown to select a variable from the form. You can also 



  5. (Optional) You can also add plain text as task identifiers. There is a 200-character limit for plain text identifiers.




  6. Select Save Changes.
  7. Go to the Queues home page and confirm that the Task Identifiers column is populated with the assigned fields and text.

 

 

Notes:

  • Deleting a variable from a form can affect Task Identifiers. If a variable is deleted from a form, make sure the Task Identifer is also updated.

  • Only four identifiers are currently supported and can be displayed in a task.

  • Identifiers are restricted to certain question types, and not all questions/items in a form can be used as Task Identifiers.

    The supported question types are:

    • Short answer
    • Date
    • Duration
    • Title/Description
    • Single Choice
    • Multiple Choice
